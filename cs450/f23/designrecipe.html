The Design Recipe

Last updated: Thu, 21 Sep 2023 11:48:53 -0400

An central concept in this course is The Design Recipe, which is a systematic way to write readable and correct high-level code (no matter what programming language you’re using).

The Textbook describe some steps of the recipe, but the official recipe steps are explained on this page.

1 Data Design

The first step of any programming task must be to determine what data the program operates on.

Since a program’s job is always to accomplish some real-world task, this data must be a represention, i.e., a data representation, of some real-world concept

More specifically, the data representation is a series of data definitions.

Data Definitions

A data definition consists of four parts

  1. a name,

  2. a description of the set of all possible values of this data,

  3. an interpretation that explains what real-world concepts are represented by this data definition,

  4. Predicate requirements updated in lecture 5 to accommodate enumerations and itemizations.

    and a predicate that accepts a single argument of any value and returns true if that input is a member of this data definition (if needed, define additional predicates for each enumeration or itemization).

Data Definition Examples
;; A TempC is an Integer
;; Interp: Represents a temperature in degrees Celsius.
(define (TempC? x) (exact-integer? x))

;; A TempF is a Real
;; Interp: Represents a temperature in degrees Fahrenheit.
(define (TempF? x) (real? x))
;; A TempK is a non-negative Integer
;; Interp: Represents a temperature in degrees Kelvin.
(define (TempK? x) (exact-nonnegative-integer? x))
Pre-defined Data Definitions

You may assume that basic Racket data types, e.g., "Integer", already have "built-in" data definitions.

In general, these data definitions correspond to the pre-defined predicates in Racket, e.g., string?.

Kinds of Data Definitions

  • basic data

  • intervals

  • enumeration

  • itemization

2 Function Design

The recipe for designing a function requires the following steps.

  1. Data Design (see above)

  2. Name of the function

  3. Signature

    The Signature specifies the number of arguments and the types of the inputs and outputs.

    It should use the Data Definitions defined in the first step.

    In particular, it should use the predicates from the Data Definition in combination with define/contract.

  4. Function Description

    The Description briefly describes, in English prose, what the function does.

  5. Examples

    Similar to the Test-Driven Development (TDD) philosophy, these must be written before any code. (Note, however, that we have a separate Tests step after the Code step.)

    One way to think about the difference between Examples and Tests is:

    • Examples help explain what a function does. They are what you would want to see in documentation and tend to be simpler.

    • Tests help verify that a function is correct. They may be more complicated and, as a result, less human-readable.

    The number of Examples needed depends on how complicated a function is. For simple functions, one example may suffice. Larger functions may require more.

    In this course, Examples should be expressed as rackunit test cases, e.g., check-equal? so they may later be used to test the code.

  6. Template step added in lecture 5

    Template

    Sketch out the structure of the function, using the data definition of the input.

    For example, for enumeration data, the Template is a cond where the clauses correspond to the enumerations (in the same order).

  7. Code

    Fill in the template (if there is one). The (remaining) code should be a combination of nested arithmetic expressions and should not contain any statements.

  8. Tests

    Tests are written after the Code step and are in addition to the Examples.

    While the tests should include the Examples, they should also include additional cases that are more complicated.

    They should follow basic software testing principles, e.g.,

    • maximum code coverage, and

    • sufficient testing of all the possible input values. One important aspect of this second point is coming up with corner cases.

    The number of Tests needed depends on how complicated a function is. For simple functions, two tests may suffice. More complicated functions may require more.

    In this class, put Tests in a separate file.

Examples of function definitions that follow the The Design Recipe:
;; c2f: TempC -> TempF
;; Converts the given Celsius temperature to an equivalent Fahrenheit one.
(check-equal? (c2f 0) 32)
(check-equal? (c2f 100) 212)
(check-equal? (c2f -40) -40)
 
(define/contract (c2f ctemp)
  (-> TempC? TempF?)
  (+ (* ctemp (/ 9 5) 32)))

;; in a separate Tests file
(check-equal? (c2f 1) 33.8)
