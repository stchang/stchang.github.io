This page gives a basic introduction to the Racket programming language.

The Racket documentation and the Textbook include many more details and will be useful resources this semester.

1 Installation

See Racket’s "Getting Started Page"

2 Programming Basics

  • All Racket files end in the .rkt extension, e.g., hw0.rkt.

  • Every Racket file (in this class) must begin with #lang racket as the first line.

    This specifies that the code that follows is written in the main racket language (Racket is actually an entire collection of different programming languages, but for this course we will stick to #lang racket only.)

  • Identifiers in Racket can use more characters than other languages typically allow. (Also, they are case sensitive!)

    Some non-traditional characters commonly used in Racket identifiers are -, ?, !, and /.

    See this page for a complete list of allowed/disallowed characters in an identifier.

  • Comment lines begin with a semicolon (;) (see commenting style in the next section).

    Alternatively, s-expressions may be commented with a "hash-semicolon" (#;).

3 Racket Style

To be readable, all code should be written according to an accepted list of style guidelines.

In this course, we will use the Racket Style Guide.

Here are a few conventions that we’ll use more frequently:

  1. Spacing: Closing parentheses do not go on their own line.

  2. Spacing: Indentation matters. When in doubt, use DrRacket’s auto-formatter.

  3. Code Organization: provides and requires go at the top of the file (provides first).

  4. Brackets have the same meaning as parens but for readability, we will use them in some situations, e.g., around cond clauses.

  5. Comments: double semicolon for full-line comments, single for partial line.

  6. Naming: Predicate (function that returns a boolean) names have a ? suffix, e.g., string?.

  7. Naming: multi-word identifiers, e.g. string-append, use - as the separator (not underscore or camelCase)

  8. No "magic numbers". Define constants instead.

  9. Naming: All constants must be named with an all-caps identifier.

4 The Design Recipe

This semester, we will learn The Design Recipe, which is a systematic way to write readable and correct high-level code.

The Textbook does describe some steps of the recipe, but for convenience, the official recipe steps will be listed here.

4.1 Data Design

The first step of any programming task must be to determine the how to represent real-world concepts as data, i.e., data representations, that the program’s code will operate on. This will likely require defining a series of data definitions.

Data Definitions

A data definition consists of four parts

  1. a name,

  2. a set representing all possible values of this data,

  3. an interpretation that explains what real-world concepts are represented by this data definition,

  4. and a predicate that accepts a single argument of any value and returns true if that input is a member of this data definition.

Examples
;; A TempC is an Integer
;; Interp: Represents a temperature in degrees Celsius.
(define (TempC? x) (exact-integer? x))
;; A TempF is a Real
;; Interp: Represents a temperature in degrees Fahrenheit.
(define (TempF? x) (real? x))
;; A TempK is a non-negative Integer
;; Interp: Represents a temperature in degrees Fahrenheit.
(define (TempK? x) (exact-nonnegative-integer? x))
Built-in Data Definitions

You may assume that basic Racket data types, e.g., "Integer", already have "built-in" data definitions. In general, these data types will already have predicates defined, e.g., string?.

4.2 Function Design

The recipe for designing functions requires the following steps.

  1. Data Design (see above)

  2. Name, Signature, and Description

    The Name is the name of the function.

    The Signature specifies the number of arguments and the types of the inputs and outputs. It should use the Data Definitions defined in the first step.

    The Description briefly describes, in English prose, what the function does.

  3. Examples

    Similar to the Test-Driven Development (TDD) philosophy, these must be written before any code.

    Examples should be expressed as rackunit test cases, so they may later be used to test the code.

  4. Code

    The code should be a combination of nested arithmetic expressions and should not contain any statements.

  5. Tests

    These tests are written after the Code and are added to the Examples.

    They should follow basic software testing principles, e.g.,

    • maximum code coverage, and

    • sufficient testing of all the possible input values.

    One important aspect of the second point above is coming up with corner cases.

Here is an example

;; c2f: TempC -> TempF
;; Converts the given celsius temperature to an equivalent Fahrenheit one.
(check-equal? (c2f 0) 32)
(check-equal? (c2f 100) 212)
(check-equal? (c2f -40) -40)
 
(define (c2f ctemp)
  (+ (* ctemp (/ 9 5) 32)))
 
(check-equal? (c2f 1) 33.8)
